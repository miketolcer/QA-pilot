Pilot Support and The Process of QA

Let’s start with some Pilot support! Take some time to read through this conversation between a customer and a QA:

Customer: “Hey, this last delivery was awful.”

QA: “I’m sorry it wasn’t a good experience, can you tell me what wasn’t good?”

Customer: “Well, first and foremost, it was ugly.”

QA: “Just so you know, the app is an early prototype, but if you have any styling requests, we can definitely talk about those.”

Customer: “Oh, I thought we were going to be stuck with that, you’re saying that your team will fix it?”

QA: “There are going to be changes to the user interface and styling, yeah. You can expect the final product to be far more polished on the UI.”

Customer: “Great, great. Well, that’s good. It still doesn’t save any of the changes we’re making.”

QA: “Yeah, that has to do with it being a prototype too, getting the database set up and hooked in is coming up on the roadmap.”

Customer: “Good, because your test data is wrong, and it’s missing fields too.”

QA: Oh, what fields are missing?

Customer: “Well, the email field’s not there, I can’t search, and I can’t log in/out either.

QA: “I see. I’ll look into it, but I know at least some of this is coming up soon. Just to make sure my notes are right, you needed the email field for the contact card, to be able to search the employee list, and want to be able to log in and out as a user?”

This conversation is in reference to this application: Employee Manager v1.0 (https://devmountain-qa.github.io/employee-manager/1.0_Version/index.html)

Take some time again to explore this site to understand it better.

Now think back to the previous conversation, take some time to answer these questions:

  1. Was the customer justified with their complaints?

  2. Did the QA do a good job responding to the customer’s complaints?

  3. Would you address them in a different way?

  4. What information would you need to adequately respond to their concerns?

Based on the conversation above, create a checklist of features that the customer would like the website to do.

Try to imagine yourself as a customer in this situation, what features would you want this employee manager site to have?

Now that you’ve thought about things from the customer perspective, let’s shift back into the QA mindset.

Here’s some documentation for Employee Manager: https://devmountain-qa.github.io/employee-manager/1.0_README.html

Let’s remember what the Process of QA is:

  1. Planning

  2. Testing

  3. Reporting

  4. Following Up

Use the documentation above to create a document that answers these questions:

  1. What are you testing?

  2. How long will you test for?

  3. Are there pieces you aren’t going to test?

  4. How do you plan on testing?

  5. How do you know when you’re done testing?

By answering these questions, you’re starting to create a plan for your testing, the first step of the Process of QA!

Now you’ll want to use that plan and take the next 30 minutes to do some testing! Take a close look at each feature in the application! This is the next phase, the actual testing!

Now based on your findings from running some tests, create a report with a list of all the issues/bug you found, but also include the positives! If you think the application is a good start, be sure to say that!

And our last phase, following up. Take the next 30 minutes to write out a report to the customer from earlier in the lab. Include the issues you’ve found, and how you’d plan to address their questions and concerns.

Push your work to your GitHub account so that your education team can review your work.
